# Simulation Guide

This file explains the current simulation in plain language.

## 1) Goal

We want to answer one question:
- Can a team switch to live sync only when it really needs it, instead of syncing all the time or never syncing?

We compare three team styles:
- `async_only`: always async
- `always_sync`: always sync
- `swarm`: mostly async, but switches to sync when the model sees strong warning signs

## 2) Core idea: meaningful signal vs normal noise

In this project:
- `signal` means a change that can trigger real rework across dependencies.
- `noise` means normal daily coordination chatter (messages/churn) that does not by itself create backward task movement.

The whole experiment is about handling signal without overreacting to noise.

## 3) Why Jira data is used

Jira gives us a realistic baseline so the simulation is not arbitrary.

We use Jira in two ways:
- To set baseline rates (realistic starting point).
- To build stress levels (q50, q90, q99) so policy differences are visible.

## 4) Two-setting strategy

We split settings into two groups.

Group A (intentionally stressed):
- `propagation.p_signal_change`
- `propagation.base_propagation`

Group B (kept realistic and mostly fixed):
- team size, task graph shape, effort distribution
- defect and delay behavior
- coordination costs
- `propagation.p_noise_change`
- policy mechanism itself

Why this split:
- Group A helps reveal differences.
- Group B keeps the scenario believable.

## 5) How the model runs (simple)

Each run has tasks with dependencies.
People work on tasks over time ticks.

At each tick:
1. Policy may switch mode (`ASYNC`/`SYNC`).
2. External changes happen:
   - `p_signal_change`: can trigger rework and cascade.
   - `p_noise_change`: adds communication load/churn.
3. Agents work on tasks.
4. Integration check runs.
5. Rework can finish and create new version updates.
6. Dependency cascades are applied.
7. Metrics and costs are recorded.

## 6) Demand and supply in this model

Per tick, the simulator computes:
- `demand`: how much coordination is needed now.
- `supply`: how much coordination the current mode provides now.
- `gap = demand - supply`.

This gap is used to label:
- high-need windows
- low-need windows

Then we measure trigger quality:
- `false_alarm_rate`: sync used in low-need windows.
- `missed_escalation_rate`: stayed async in high-need windows.

## 7) Main metrics (paper-friendly)

- `bounce_rate`: how often tasks moved backward.
- `eta`: output quality per coordination effort.
- `false_alarm_rate`: unnecessary sync.
- `missed_escalation_rate`: missed sync when needed.

Extra table:
- fixed-cost signal capture compares swarm vs async at similar coordination cost.

## 8) Current swarm default behavior

Current default in `config.yaml`:
- trigger based on signal-to-noise ratio
- gap gate enabled
- cooldown enabled (`cooldown_ticks`) to reduce rapid mode flapping

This means swarm does not switch instantly on every small blip.

## 9) Stress regimes

Default ladder:
- low: q50-like pressure
- near-threshold: q90-like pressure
- high: q99-like pressure

This comes from Jira-derived quantiles and is encoded in `stress_ladder`.

## 10) Files to read after a run

Always generated by `s2n run`:
- `results/runs_summary.csv`
- `results/cascades.csv`
- `results/timeseries.csv`
- `results/events.csv` (unless `--no-events`)

Always generated by `s2n resources`:
- `results/resources/mode_regime_summary.csv`
- `results/resources/signal_capture_fixed_cost.csv`

Optional analysis tables (generated by extra sweep scripts):
- `results/resources/near_threshold_micro_sweep.csv`
- `results/resources/swarm_profile_tradeoff.csv`
- `results/resources/trigger_tuning_precision.csv`

## 11) Commands

```bash
source .venv/bin/activate
s2n run --config config.yaml --out results
s2n resources --in results --out results/resources
```
